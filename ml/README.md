# RMUC 对局预测实验

当前网页默认使用一个真实训练的时序 Transformer：给定最近 5 秒的全场状态，预测一辆仍存活机器人的未来位置。旧 MLP 检查点仍保留作可复现基线。

模型同时输出 `1 / 3 / 5 / 10 / 15` 秒后的位置。输入包含：

- 双方 12 辆移动机器人最近 0、1、3、5 秒的位置与血量
- 双方基地、前哨站血量和剩余金币
- 目标机器人的速度、朝向、弹药变化、易伤状态和类型
- 当前比赛进度

比赛进度固定按规则时长 420 秒计算，不使用赛后才知道的实际结束时间。

蓝方样本会旋转 180° 到红方视角，因此模型学习的是阵营无关的战术规律。训练、验证、测试按整场比赛分组；同一场 BO3/BO5 的不同局不会跨集合，避免测试分数虚高。

## 训练时序 Transformer（当前默认）

模型把 `0 / -1 / -3 / -5` 秒的全场状态投影成 4 个战场 token，再加入 1 个目标机器人上下文 token，经过 3 层、4 头、252,394 参数的 Transformer Encoder 输出未来 `1 / 3 / 5 / 10 / 15` 秒位置残差。上下文包含目标队伍、对手队伍和最近 1/3/5 秒受击量，因此能区分各队在受击前后的部署变化。它是 PyTorch `nn.TransformerEncoder` 的真实多头自注意力训练，不是把 MLP 改名，也不是大语言模型。

```bash
python3 ml/train_trajectory_transformer.py
python3 ml/export_browser_transformer.py
```

默认用全部 613 局，按系列赛隔离为 431 / 94 / 88 局，最多取 30 万训练样本。固定分组种子优先扩大每支队伍在验证/测试中的覆盖；同一场 BO3/BO5 仍不会拆到不同集合。输出为：

```text
ml/artifacts/trajectory_transformer.pt
ml/artifacts/trajectory_transformer.metrics.json
docs/data/models/trajectory_transformer.json
docs/data/models/trajectory_transformer.bin
```

非补给区的静止样本保留 70% 权重，用于学习远程吊射、架点和防守；补给区静止且 15 秒内位移不足 0.75 m 的权重降为移动样本的 `0.70 × 0.10 = 7%`。最近受击样本提升到 1.5 倍，帮助模型学习“受击后离开阵位”，而不是把所有静止一律当坏样本。快速验证命令：

```bash
python3 ml/train_trajectory_transformer.py \
  --epochs 2 \
  --max-games-per-split 10 \
  --max-train-samples 5000 \
  --max-val-samples 1000 \
  --max-test-samples 1000
```

正式训练不要设置 `--max-games-per-split`。训练完成后必须执行导出脚本，并运行 `python3 -m unittest ml.test_trajectory_transformer` 验证浏览器 JavaScript 与 PyTorch 推理数值一致。

## 旧 MLP 基线（保留）

在项目根目录运行：

```bash
python3 ml/train_trajectory.py
```

默认使用 CUDA（存在时），最多抽取 30 万训练样本。模型和完整评估报告分别保存到：

```text
ml/artifacts/trajectory_mlp.pt

```

快速验证数据管道可以缩小样本和轮数：

```bash
python3 ml/train_trajectory.py \
  --epochs 2 \
  --max-train-samples 20000 \
  --max-val-samples 5000 \
  --max-test-samples 5000
```

报告会把模型与两个基线比较：

- `stationary`：假设机器人停在原地
- `constant_velocity`：按最近 3 秒速度直线外推
- `learned_mlp`：本模型

位置误差之外，`zone_accuracy` 把 28 m × 15 m 场地划成 4 m × 3 m 的战术格，衡量“下一步会去哪一片区域”。

## 当前 Transformer 结果

当前检查点使用 30 万训练样本，在按整场比赛隔离的 88 局、67,429 条测试样本上评估。下表是平均位置误差（米）；“移动中”表示目标最近 3 秒速度不低于 0.15 m/s。

| 预测跨度 | 原地基线（全量） | Transformer（全量） | 原地基线（移动中） | Transformer（移动中） |
| ---: | ---: | ---: | ---: | ---: |
| 1 秒 | 0.405 | 0.326 | 0.758 | 0.535 |
| 3 秒 | 0.943 | 0.780 | 1.777 | 1.324 |
| 5 秒 | 1.370 | 1.173 | 2.482 | 1.922 |
| 10 秒 | 2.156 | 1.907 | 3.634 | 2.913 |
| 15 秒 | 2.686 | 2.405 | 4.315 | 3.503 |

Transformer 的 5 秒移动目标 4 m × 3 m 战术格命中率为 51.88%。同济大学有 4 局训练、4 局验证、2 局独立测试；独立测试中，英雄未受击时模型预测的 10 秒平均位移为 0.357 m、守点率 94.6%，近期受击后为 1.030 m、守点率 66.7%。完整指标见 `artifacts/trajectory_transformer.metrics.json`，条件行为由 `test_tongji_hero_leaves_anchor_more_after_damage_on_held_out_games` 回归测试保护。

```bash

```

也可以只看一辆车：

```bash
python3 ml/predict_trajectory.py \
  --game-id 1780037479635 \
  --second 200 \
  --side 红 \
  --robot-type 哨兵
```

如果预测时刻之后的真实帧仍在该局数据中，输出会一并给出真实坐标和误差，便于回放对照。

这还是行为预测模型，不是可做反事实推演的完整环境。后续要训练决策策略，应先在它之上增加事件/伤害预测、资源状态转移和不确定性分支，再做离线模仿学习或保守离线强化学习。

- 每队雷达能力由区域赛反制事件统计；空中最多被反制 5 次、每次只锁定发射机构 45 秒，不会被击毁或强制降落，第 4/5 次可以双倍金币解除；

### 怎么训练和修正沙盘行为

不要直接改 `docs/data/models/*.json`，它们都是构建产物。正确的分层是：

- **硬约束**：场地实体边界、规则伤害和哪些兵种有资格通过，分别在 `analysis/terrain_crossing_points.py`、`ml/export_browser_navigation.py` 和规则模型中维护，不交给概率模型违反。
- **地形能力标签**：`analysis/manual_terrain_capabilities.csv` 只写“某校×某兵种会/不会过”的人工监督标签；其余样本由完整穿越轨迹自动标注。
- **队伍行为标签**：`analysis/manual_team_behavior_labels.csv` 用于数据没有显式字段的监督信息，例如英雄整机类型、远/近程风格、确认的前哨任务角色。队名只出现在训练标签和模型中，不得在 JS 引擎写 `if (队名)` 特判。
- **44 队覆盖门禁**：每次构建后测试会验证 44/44 都拥有上述画像与地形动作模型，并检查运行时引擎不包含任何学校名。当前 44 队行为签名全部不同；样本不足只能在单个维度上回退，不会把整支队替换成统一模板。
- **可学习过程**：`ml/train_terrain_motion_profiles.py` 从裁判“飞坡”事件前后的 1 Hz 轨迹学习每校×兵种的直线助跑距离、对位概率、停顿概率/时长和三段加速曲线；助跑距离取终点为飞坡事件时刻的最后一个轨迹位移段并限制在 1.0–2.8 m。同时检测完整从一侧穿到另一侧的 B3/R3 轨迹，将上台阶和下台阶分开学习入射偏角、横向偏移和直行概率。单个方向的样本少于 5 次时才回退到该方向的全局分布。
- **路径形状约束**：B1/R1 只能沿飞坡通道中心线通过，正飞坡和有反飞坡证据的兵种都按“助跑点→坡口中心→出口中心”生成共线航点；确认能直接跃上中央高地的兵种也必须沿所选边界的法线完成“直线助跑→坡唇→落点”。这些几何参数导出到 `terrain_navigation.json` 的 `routing.terrain_route_profiles`，快速宏观预测和完整沙盘共用同一个 `terrain-router.js`。
- **补给撤退**：补弹不再按直线距离选择基地/前哨/补给区，而是比较实际可达路线长度；直达失败时先撤到己方半场中继点再规划第二段。若当前位置与己方区域在已确认能力图中确实不连通，模型不会再假装“正在回补给”却原地等待，而会就地转点 3 秒后重试。对应策略参数也在 `routing.terrain_route_profiles.service_return`。
- **火力概率**：逐队发弹数与对手受击事件生成 `accuracy_models`；每局先在该队区间内抽样基础命中率，每一发再做伯努利随机判定。英雄对机器人/前哨/基地的 42mm 主模态分开统计，防御减伤后的小于 200 数值不会反向学成原始弹丸伤害。

`manual_terrain_capabilities.csv` 的 `ability` 可填 `central_highland_step`、`road_step`、`fly_ramp`、`rough_road`、`road_tunnel`、`highland_tunnel`、`slope_43`、`trapezoid_highland_step` 或 `central_highland_400mm_jump`，`label` 只能是 `confirmed` / `rejected`。`manual_team_behavior_labels.csv` 当前支持 `outpost_assault_role`、`hero_archetype_default`、`engagement_style` 和 `robot/outpost/base_damage_per_hit`。每次改 CSV 后都必须重新运行下面的构建链。

完整重训顺序：

```bash
python3 analysis/team_terrain_capabilities.py
python3 ml/train_terrain_motion_profiles.py
python3 ml/export_browser_navigation.py
python3 ml/build_full_simulation_data.py
python3 -m unittest analysis.test_team_terrain_capabilities ml.test_browser_navigation ml.test_match_simulator
```

想加新的跨地形属性时，先在 `terrain_crossing_points.py` 定义几何门，再在 `team_terrain_capabilities.py` 定义正负样本口径；如果它还包含“直线助跑→对位→停顿→加速→落地”时序或“入射角→横向漂移”轨迹特征，再给 `train_terrain_motion_profiles.py` 增加事件窗口，最后由 `export_browser_navigation.py` 导出到浏览器模型。中心线、边界法线等不能被概率违反的条件放进 `terrain_route_profiles`；从轨迹能估计的距离、概率和速度放进每校×兵种的 `terrain_motion_profiles`。

### 快速宏观推演

上方完整沙盘与下方快速推演共用红蓝学校选择，任一处修改都会同步另一处并重开对应推演。快速模型同样维护基地护甲状态，闭甲阶段不会把 15 秒历史混合火力整段当作满额基地伤害；飞镖的连续混合值会还原为基地 200/300/625/1000 或前哨 750 的离散单发伤害，并把档位写入回合日志。

网页底部的“对局模拟”把一局 420 秒切成 28 个 15 秒回合。每队每个回合会从真实比赛样本中抽取并融合：

- 基地、前哨站和地面机器人的伤害；
- 17mm、42mm 与飞镖伤害构成；
- 飞镖命中/开闸窗口；
- 己方与敌方堡垒附近的驻留时间；
- 能量机关增益和官方记录的地形动作。

网页默认展示 TDT 对 RPS，可逐回合落子、自动演算单局，或批量模拟 500 局。跨赛区对阵属于反事实推演，批量结果是用于比较战术风格的实验分布，不是官方胜率。

底层静态棋谱由区域赛 SQLite 数据生成：

```bash
python3 ml/build_match_simulation_data.py
```

输出文件为 `docs/data/models/match_simulation.json`。堡垒控制目前通过存活且具备占领资格的英雄、步兵和哨兵在堡垒中心附近的驻留秒数推断，工程和空中不计入；事件表中没有独立的裁判占领事件。

## 操作手战术推理

`tactical_inference.py` 把坐标预测转成客户端可直接显示的内容：

- 5/10/15 秒候选战术区域，10 秒为主目标；
- 预计经过的 B/R 地形入口和路线摘要；
- 对照人工确认/官方事件/轨迹证据的能力风险提示；
- 全场状态模型与 44 队的学校×兵种×区域×比赛阶段先验融合。

构建/更新战队先验：

```bash
python3 ml/build_tactical_priors.py
```

回放某一时刻的操作手输出：

```bash
python3 ml/tactical_inference.py \
  --game-id 1778680444314 --second 200 \
  --side 红 --robot-type 步兵3
```

本地 HTTP 服务：

```bash
python3 ml/tactical_api.py --host 127.0.0.1 --port 8765
```

`POST /infer` 只需提供当前 `second`、红蓝学校信息和最近 5 秒帧。帧内每个机器人使用导出数据的紧凑数组格式：

```text
[id,type,side,hp,max_hp,x,y,yaw,ammo17,ammo42,coins,vulnerable]
```

推理器使用 MC dropout 生成候选分布；`display_confidence` 还会乘以独立测试集在对应时距上的历史战术格命中率。它适合做提示强度，不应解释为严格的贝叶斯概率。

### 在比赛回放中演示

直接启动当前 PyQt 回放客户端：

```bash
python3 app.py
```

选择赛区、场次和局次后，点击时间轴右侧的“开启预测”。地图会显示：

- 红/蓝虚线：从当前位置到未来 10 秒的粗轨迹；
- 圆环：10 秒主目标点；
- 标签：兵种、目标战术区域和显示置信度；
- 右下状态：当前输出的存活地面机器人数和单次推理耗时。

拖动时间轴或正常播放时会自动请求新的整秒预测。模型在后台线程运行，不阻塞时间轴；阵亡机器人不绘制未来路线。
